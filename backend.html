Chain reference model
Presence Protocol currently anchors signed BLE encounter evidence as Cardano Leios testnet metadata transactions. These are public chain references to off-chain verified encounter payloads, not validated ZK proofs. ZK claim validation is planned for the Midnight layer.

End-to-end pipeline

1

BLE discovery

Two devices running the Presence Protocol app detect each other via Bluetooth Low Energy. A GATT handshake is initiated within seconds of proximity.

2

Dual signature verification

Each device signs the encounter payload with an ephemeral P-256 key. Both signatures are verified before any reward is issued. Unsigned or mismatched encounters are silently dropped.

3

Ticket generation & ledger credit

A deterministic encounterId is derived from the handshake payload. CPOP tokens are issued via the harmonic issuance engine and credited to the local ledger.

4

Anchor POST to backend

SyncCoordinator serialises a CpopAnchorPayload and POSTs it to the backend API. The backend deduplicates by anchorHash and stores the record.

5

Leios testnet submission

Anchor payloads are submitted as metadata transactions to the Cardano Leios testnet (Dijkstra era, network magic 164) via cardano-cli dijkstra transaction build/sign/submit.

6

Midnight ZK validation

Planned: encounter proofs submitted to Midnight Network for zero-knowledge validation. Private claim processing via Compact smart contracts. Access pending Midnight Foundation DApp Builder approval.
